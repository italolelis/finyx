<known_limitations>
The Claude Code `@.finyx/profile.json` directives in `<execution_context>` blocks load the file LITERALLY at skill startup, before any bash runs. There is no way to make `@` follow `$FINYX_PROFILE` dynamically. Two consequences:

1. If user has `$FINYX_PROFILE` pointing to `/elsewhere/profile.json` and no project-local `.finyx/profile.json` exists, the `@` include will silently no-op (Claude Code does not error on missing @-includes — it just omits the reference).

2. The actual profile content the skill operates on comes from the bash `Read` step that follows the gate check (e.g., "Read `.finyx/profile.json`" instructions in each skill). These Read instructions must be updated to use the resolved path: `Read $(${CLAUDE_SKILL_DIR}/../../scripts/resolve-profile.sh)` or equivalent — but doing this for ALL "Read .finyx/profile.json" mentions across 19 files balloons scope beyond what was requested.

**Resolution for this plan:** We replace the gate checks (Pattern A/B/C/D) and add a one-line note to each skill's `<execution_context>` explaining that `@.finyx/profile.json` is a fast-path optimization for project-local profiles only; the resolver-determined path is authoritative. Subsequent `Read` instructions in each skill's `<process>` already say "already loaded" — those references implicitly trust the resolved path because the gate check now exports `$PROFILE_PATH`. Updating the per-phase Read instructions to reference `$PROFILE_PATH` instead of the hardcoded `.finyx/profile.json` is in scope ONLY for the gate-check-adjacent Read instruction (the next 1–3 lines after the gate). Deep-rewriting every Read mention is OUT OF SCOPE; doc-reader.md and portfolio.md (which read profile.json multiple times in different phases) are explicitly skipped — they live in sub-skills and currently work fine with project-local; relocating them is a separate task.

Create `scripts/resolve-profile.sh` as a POSIX-compatible bash script (shebang `#!/usr/bin/env bash`, `set -euo pipefail`). Make it executable (chmod +x).

**Interface contract:**

Two invocation modes:

1. **Default (read mode):** `./scripts/resolve-profile.sh`
   - Resolves the path to an EXISTING profile.json using 4-tier precedence
   - On success: prints absolute path to stdout, exits 0
   - On failure: prints actionable error to stderr, exits with documented non-zero code

2. **Write-target mode:** `./scripts/resolve-profile.sh --write-target`
   - Resolves the path where a NEW profile.json SHOULD be created
   - Does NOT require the file to exist
   - On success: prints absolute path to stdout, exits 0
   - On failure (e.g., env var path's parent dir is unwritable): exits non-zero with stderr error

**Read mode precedence (--write-target uses the same precedence but skips the existence check on the chosen tier):**

```
Tier 1: $FINYX_PROFILE env var
  - If set and non-empty:
    - In read mode: file MUST exist → if missing, error exit 2 with message: "FINYX_PROFILE points to '$FINYX_PROFILE' but file does not exist. Either create the file, unset FINYX_PROFILE, or fix the path."
    - In write-target mode: parent dir MUST be writable → if not, error exit 3
    - Output: absolute path of $FINYX_PROFILE (resolve via realpath -m or python -c if available, else printf as-is when already absolute)

Tier 2: ~/.config/finyx/config.json with key "profile_path"
  - If file exists AND contains valid JSON AND has top-level key "profile_path":
    - Extract value (use python3 -c 'import json,sys; d=json.load(open(sys.argv[1])); print(d.get("profile_path",""))' as primary; fallback to grep+sed if python3 absent)
    - Expand ~ to $HOME
    - In read mode: file MUST exist → if missing, error exit 2 with message: "config.json profile_path points to '<resolved>' but file does not exist. Fix ~/.config/finyx/config.json or remove the profile_path key."
    - In write-target mode: parent dir MUST be writable → if not, error exit 3
    - Output: absolute path

Tier 3: ./.finyx/profile.json (project-local)
  - In read mode: if exists → output absolute path (use $(pwd)/.finyx/profile.json)
  - In write-target mode: if .finyx/ exists OR (.git OR package.json OR Makefile in CWD) → output $(pwd)/.finyx/profile.json

Tier 4: ~/.finyx/profile.json (global fallback)
  - In read mode: if exists → output $HOME/.finyx/profile.json; if not → error exit 1 with full diagnostic listing all 4 tiers and what was checked
  - In write-target mode: always selected as final fallback → output $HOME/.finyx/profile.json
```

**Exit codes:**
- 0 = success
- 1 = no profile found across all tiers (read mode only)
- 2 = configured override (env var or config.json) points to missing file (read mode)
- 3 = write target's parent dir is not writable (write-target mode)
- 4 = config.json exists but is malformed JSON (both modes — fail loudly per known_constraints)

**Error message format:** All errors go to stderr with prefix `finyx-resolve-profile:`. Always include WHICH tier failed and HOW to fix it. Example for exit 1:

```
finyx-resolve-profile: ERROR: No financial profile found.
  Checked (in precedence order):
    1. $FINYX_PROFILE env var          [not set]
    2. ~/.config/finyx/config.json     [not present]
    3. ./.finyx/profile.json           [not present]
    4. ~/.finyx/profile.json           [not present]
  Run /finyx:profile to create a profile.
```

**Implementation notes:**
- Use `command -v python3` to test for python3; if absent, fall back to a regex-based JSON extraction limited to top-level string values for the `profile_path` key only. Never attempt to fully parse JSON in shell.
- Use `realpath -m` if available, else `realpath`, else implement minimal absolute-path resolution (`case "$path" in /*) echo "$path";; *) echo "$(pwd)/$path";; esac`).
- Detect `--write-target` as the only flag accepted. Any other arg → exit 64 with usage banner. `--help` / `-h` → print usage, exit 0.
- Quote all variable expansions. Use `[ -n "${VAR-}" ]` to be safe with `set -u`.
- Keep total LOC under ~150 lines including comments and the help banner.

## Configuration

By default, Finyx reads your financial profile from `./.finyx/profile.json` (project-local) or `~/.finyx/profile.json` (global fallback). You can override this via either an environment variable or a user-level config file.

### Profile path precedence

Finyx resolves `profile.json` using this 4-tier precedence (first match wins):

| # | Source | Notes |
|---|--------|-------|
| 1 | `$FINYX_PROFILE` env var | Highest priority. Must point to an existing file. |
| 2 | `~/.config/finyx/config.json` `profile_path` key | User-level default. |
| 3 | `./.finyx/profile.json` | Project-local profile in current working directory. |
| 4 | `~/.finyx/profile.json` | Global fallback. Used when none of the above exist. |

If `$FINYX_PROFILE` or the `config.json` `profile_path` is set but the file does not exist, Finyx fails loudly with a clear error rather than silently falling through to the next tier. This prevents silent desync.

### Example: per-shell override

```bash
export FINYX_PROFILE="$HOME/Documents/finances/personal-profile.json"
# All /finyx:* commands in this shell now use the file above.
```

### Example: user-level default

```bash
mkdir -p ~/.config/finyx
cat > ~/.config/finyx/config.json <<'JSON'
{
  "profile_path": "/Users/me/Documents/finances/personal-profile.json"
}
JSON
```

### Working artifacts stay project-local

Only `profile.json` is relocatable. Per-project working artifacts continue to live under `./.finyx/` of the current working directory regardless of where `profile.json` is stored:

| Path | Purpose |
|------|---------|
| `./.finyx/insights-config.json` | Allocation mapping persisted by `/finyx:insights` |
| `./.finyx/research/` | Location and market research output |
| `./.finyx/analysis/` | Per-location analysis (UNITS, RANKED, SHORTLIST) |
| `./.finyx/output/` | Generated briefings |
| `./.finyx/STATE.md` | Per-project workflow state |

This keeps multiple investment projects independent while letting you share one financial profile across all of them.

For implementation details, see [`scripts/README.md`](scripts/README.md).
```
